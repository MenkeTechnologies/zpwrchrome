what it is

A drop-in replacement for the Go binary that the browserpass browser extension talks to via Chrome / Firefox native messaging — plus three additive actions that browserpass-extension doesn't call but are wired into zpwrchrome: TOTP retrieval, whole-store fuzzy + substring search, and a multi-segment HTTP/HTTPS download manager.

Single static binaryPure-Rust dep tree: serde, serde_json, ureq with rustls. No aria2, no system OpenSSL, no Go toolchain at runtime.
Wire-compatible with upstreamEvery action documented in browserpass-native's PROTOCOL.md v3.1.2 is implemented. Error codes 10–32 pin to the same integers; exit code = error code; version reported as 3.1.2 (packed int 3_001_002).
Strict 1:1 port disciplineEvery src/ported/ file mirrors a single upstream Go file. Every Rust fn carries /// Port of <name>() from <go_file>:<line>. Every Go comment carries over with // go:NN citation. Identifiers preserved verbatim (DetectGpgBinary, MakeConfigureResponse).
OTP extShells pass otp <entry> with PASSWORD_STORE_DIR set to the matching store. Returns the current TOTP code.
Search extHost-side fuzzy + substring scoring across every configured store. Faster than client-side fzf for large stores.
Segmented downloads extN-segment Range GETs with retry / resume. Chrome cookies + User-Agent forwarded. File-state JSON tracker.
Audit-friendly identifiersGo's PascalCase / camelCase preserved. File-level #![allow(non_snake_case)] makes this an explicit, documented decision rather than a style accident.
Drift pinned by teststests/ported_errors.rs pins every error code to its PROTOCOL.md value. tests/ported_version.rs pins version triple + packed int. tests/ported_integration.rs exercises every action end-to-end against the compiled binary.

extension actions additive

These are additive — browserpass-extension never sends them, so wire compatibility with upstream is preserved. They live under src/extensions/.

actionbehaviorwire shape
otp Shells pass otp <entry> with PASSWORD_STORE_DIR set to the matching store. Returns the current TOTP code. {action:"otp", storeId, file, settings} → {code:"123456"}
search Host-side fuzzy + substring scoring across every configured store. Faster than client-side fzf for large stores. {action:"search", settings, echoResponse:"<query>"} → {matches:[{store, path}, …]}
download-start / download-status / download-cancel Multi-segment HTTP/HTTPS downloader. N-segment Range GETs with retry / resume. Chrome cookies + UA forwarded for logged-in URLs. {action:"download-start", url, cookie, segments} → file-state JSON; {action:"download-status", id}; {action:"download-cancel", id}

install

  1. cargo install browserpass-host-rs
  2. Register the binary for an extension ID: browserpass-host-rs --install <ext-id> — get the ext-id at chrome://extensions with Developer mode enabled.
  3. pass support requires GPG configured locally. otp requires pass-otp in $PATH.
  4. Used by zpwrchrome (a Chrome extension); also a drop-in for the upstream browserpass-extension with the OTP / search / download actions silently ignored.

Alternative: clone and cargo build --release, then install the resulting binary at the upstream-expected path ($XDG_CONFIG_HOME/<browser>/NativeMessagingHosts/com.github.browserpass.native.json). The --install flag does this automatically per browser.

port discipline reference

The src/ported/ tree is a 1:1 Rust mirror of the upstream Go source. Discipline borrowed from the zshrs PORT.md rules, adapted to a Go→Rust port:

Stem mirrorEvery file under src/ported/ mirrors a single upstream Go file by stem and relative subpath. errors/errors.go → src/ported/errors/errors.rs.
Doc-comment citationEvery Rust fn carries /// Port of <name>() from <go_file>:<line>.
Inline-comment citationEvery Go inline comment carries over to the Rust port with a // go:NN line-number citation on the corresponding Rust statement.
Identifiers verbatimGo's PascalCase / camelCase identifiers preserved (DetectGpgBinary, MakeConfigureResponse, parseRequestLength). File-level #![allow(non_snake_case)] documents the choice.
Local-name verbatimLocal variable names match Go's (gpgPath, normalizedStorePath, parentDir, responseData).
No invented helpersThe two recursive store walkers inside list.rs and tree.rs are inlined private fns at the call site — they replace external Go deps (mattn/go-zglob, mattn/go-zglob/fastwalk), not Go-source fns.

example request / response

Native-messaging frame: little-endian 4-byte length prefix + UTF-8 JSON. Same wire format as upstream. 

# list every entry in the "personal" store
{"action":"list","settings":{"stores":[{"id":"personal","path":"~/.password-store"}]}}

# → ok response: tree-shaped list
{"status":"ok","data":{"personal":["work/gmail","work/slack","banking/chase"]}}

# fetch + decrypt one entry
{"action":"fetch","settings":{"stores":[...]},"storeId":"personal","file":"work/gmail.gpg","echoResponse":""}

# → ok response: { plain text from gpg --decrypt }
{"status":"ok","data":{"contents":"hunter2\nuser: jacob@menketechnologies.com\notpauth://totp/..."}}
